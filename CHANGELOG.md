## [0.4.0] - 2026-06-10

**Form Over Subtraction Edition.** This release rewrites the form of
constitution and the seven baseline knowledge documents so each
artifact has *one authoritative shape*, drawn from the de-facto
standard for that artifact category. Every artifact in the project
still maps to one of the five subsystems established in 0.3.0; what
changed is the template each command fills.

The diagnosis: 0.3.x's constitution and baseline output were
abstract because the templates *allowed* abstraction. 0.4.0 forces
each section to one specific form (table, numbered runbook, anchor,
etc.) sourced from the standard appropriate to that artifact â€”
spec-kit for constitution, arc42+C4 for architecture, BIZBOK+BABOK
for business, ISO/IEC 11179 for glossary, Google SRE for setup,
etc. LLMs writing into a tightly-formed slot can't smuggle in
abstract prose.

### Added

- **`harness.core.content_quality`** â€” content-quality checker for
  baseline / constitution artifacts. Detects path:line citations,
  required headings, structure-form (tables and numbered steps),
  spec-kit version line, and Sync Impact Report. Exposed to
  `harness doctor` as new sub-rows under Instructions and Scope.
- **`harness doctor` content-quality checks**:
  - Constitution: verifies spec-kit `**Version**:` line, presence
    of `## Governance`, and Sync Impact Report comment at top.
  - Each baseline doc: cite-density floor (â‰¥1 path:line cite per
    200 words averaged across the file), required-sections
    presence, structure-form (single-table for `glossary.md`,
    numbered steps for `setup-and-verify.md` and
    `architecture.md`).

### Changed

- **`/hx-constitution`** rewritten to follow the GitHub spec-kit
  standard exactly. Output structure mirrors spec-kit's
  `constitution-template.md` (Core Principles â†’ 2 generic sections
  â†’ Governance â†’ version line). Execution flow follows spec-kit's
  8-step `commands/constitution.md` source. `(NON-NEGOTIABLE)` is a
  title suffix per spec-kit usage. Versioning is spec-kit semver
  (MAJOR for principle removal, MINOR for added principles or
  sections, PATCH for clarifications). `[enforceable]/[inferential]`
  tags removed from the constitution â€” those are project-
  implementation concerns and live in `AGENTS.md` `## Conventions`
  and `.harness/knowledge/conventions.md`.
- **`/hx-baseline`** restructured around an explicit
  **MEASURE â†’ SYNTHESIZE â†’ VERIFY** pipeline. MEASURE runs
  per-stack tools (pyreverse / madge / jdeps / cargo modules /
  go-callvis / syft / git churn / git log incident grep) into a
  scratch dir; SYNTHESIZE has the LLM organise tool output into
  the seven docs with mandatory path:line citations; VERIFY runs
  the new `harness doctor` content-quality checks.
- **Baseline 7-doc templates rewritten by authoritative form**:
  - `architecture.md`: arc42 + C4 + new `## Invariants` table.
  - `tech-stack.md`: Thoughtworks Tech Radar cards.
  - `product.md`: Cohn user-story + Christensen Jobs-to-be-Done.
  - `business.md`: **architecture-neutral** â€” 7 fixed sections
    grounded in BIZBOK (OMG) + BABOK v3 (IIBA) + OKR/SLO. DDD /
    Event Storming / Domain Storytelling vocabulary removed; those
    belong in `architecture.md` if the project uses that style.
  - `conventions.md`: explicit `## Reference Exemplars` anchor
    table per Thoughtworks "Anchoring AI to a reference application".
  - `glossary.md`: single ISO/IEC 11179-style table.
  - `setup-and-verify.md`: Google SRE runbook â€” First Run
    Sequence + Verification Targets + Common Failures + Environment
    Matrix.

### Migration (0.3 â†’ 0.4)

`harness init --force` on a 0.3.x project preserves the existing
`.harness/memory/constitution.md` as
`.harness/memory/constitution.md.0.3.bak` if it predates the
spec-kit format (no Sync Impact Report and no `**Version**:` line).
The freshly-formed file is regenerated by re-running
`/hx-constitution`. Existing baseline `*.md` files are not
overwritten; doctor flags content-quality drift and recommends
`/hx-baseline --refresh` per file.

## [0.3.0] - 2026-06-10

**Harness OS Edition.** This release re-grounds harness-scaffold on
the **5-subsystem operating-system model** for AI coding agents
distilled from walkinglabs *learn-harness-engineering* and Anthropic's
*Effective Harnesses for Long-Running Agents*. Every artifact in a
harness project now belongs to exactly one of:

| Subsystem | Minimum artifact |
|---|---|
| Instructions | `AGENTS.md` |
| State | `feature_list.json` per spec |
| Verification | `init.sh` (calls `verify.sh`) |
| Scope | `depends_on` graph in `feature_list.json` |
| Lifecycle | `session-handoff.md` per spec |

If a future addition can't be slotted into one of these five, it
doesn't belong in the harness.

### Added

- **`AGENTS.md.j2`** rewritten as a superset of the agentsmd standard
  with five top-level sections â€” Instructions / State / Verification
  / Scope / Lifecycle. The Output Language Contract is now `Â§ 1.1`;
  the Resume / Advance / Handoff session contract is fixed under
  `Â§ 5.1`. Existing playbooks, slash-commands, and adapters point at
  these section anchors.
- **`init.sh` / `init.ps1`** at project root â€” the single command an
  agent runs at session start. Prints project summary, runs
  environment health checks, calls
  `.harness/scripts/<sh|ps>/verify.sh`. On exit it writes
  `.harness/.last-verify.json` (status + timestamp) so other tools
  can prove verify recently passed.
- **`feature_list.json` per spec** â€” machine-readable State source of
  truth, generated by `/hx-tasks` alongside `tasks.md`. Schema ships
  at `.harness/templates/feature_list.schema.json`. Status enum:
  `not-started` â†’ `in-progress` â†’ `passing`. Agents may flip
  `status` and write `evidence` only â€” never edit `id`, `name`,
  `verification`, or `depends_on`.
- **`harness state` CLI** â€” three subcommands:
  - `harness state show [spec-id]` renders the feature table.
  - `harness state next` prints the next executable feature
    (depends-on resolved, status != passing).
  - `harness state pass <feat-id> --evidence <ref>` flips a feature
    to `passing`. Refuses unless `init.sh` ran green within the last
    five minutes (read from `.harness/.last-verify.json`); `--force`
    available for explicit overrides.
- **`session-handoff.md` per spec** â€” six fixed sections (Current
  Objective Â· Completed Â· Verification Evidence Â· Files Changed Â·
  Decisions Â· Blockers Â· Next Session Startup). `/hx-implement`
  overwrites it at session end; `/hx-next` reads it first when
  routing. Cross-spec timeline still lives in `.agent/progress.md`
  (append-only).
- **12-Factor Agents principle pack** at
  `.harness/principle-packs/12-factor-agents.md` (12 cards,
  `P-12-1` â€¦ `P-12-12`, attributed to HumanLayer). Auto-included by
  `/hx-constitution` synthesis when the project profile is
  `brownfield` or `vibecode`.
- **`vibecode` profile** â€” third profile (alongside `brownfield`,
  `greenfield`) for projects inheriting AI-generated codebases.
  Picks the same defaults as `brownfield` plus the 12-Factor pack
  and signals the constitution synthesizer to bias toward
  ratcheting-down rules rather than greenfield enforceables.
- **`harness doctor` rewrite**: top-level subsystem health table
  (Instructions Â· State Â· Verification Â· Scope Â· Lifecycle) with
  per-row OK / WARN / FAIL roll-ups. New checks: `init.sh` present
  and executable, `feature_list.json` schema-valid + cycle-free, ID
  parity vs `tasks.md`, `session-handoff.md` freshness (7-day
  window), `.last-verify.json` status + age.

### Changed

- **`/hx-tasks`** now emits both `tasks.md` (human narrative) and
  `feature_list.json` (machine truth); both must agree on task IDs.
- **`/hx-implement`** rewritten as a Resume â†’ Advance â†’ Handoff
  loop: read `feature_list.json` and `session-handoff.md` first,
  pick one feature with `harness state next`, run `./init.sh` to
  verify, flip status with `harness state pass`, overwrite
  `session-handoff.md` at session end. Old upstream-flaw rollback
  + verify-failure circuit-breaker semantics preserved.
- **`/hx-next`** reads `feature_list.json` and `session-handoff.md`
  before flow analysis; the handoff's "Next Session Startup" often
  answers the routing question directly.
- **AGENTS.md** now declares itself a superset of the
  `agentsmd/agents.md` standard (standard sections under
  `## 1. Instructions`).

## [0.2.0] - 2026-06-08

A pivot on how the scaffold handles multilingual output. Instead of
generating per-locale template variants at `harness init` time, the
scaffold now produces a single English source-of-truth and lets each
`/hx-*` agent decide deliverable narrative language at runtime,
guided by an explicit Output Language Contract.

### Added

- **`harness.core.i18n`** â€” runtime message tables (zh / en) covering
  every CLI-emitted string: `harness init` progress prints, summary
  table headings and rows, the "Next steps" panel, the interactive
  `questionary` agent picker, and `harness doctor` diagnostics. CLI
  output now honors `--output-lang` (default `zh`) and `doctor`
  reads it from `.harness/config.toml`.
- **`## Output Language Contract`** â€” new top-of-AGENTS.md section,
  rendered with the live `output_lang` value, that tells every
  `/hx-*` agent how to write deliverables: prose follows
  `output_lang`; headings, frontmatter, Provenance blocks, RFC 2119
  keywords (MUST / MUST NOT / SHOULD / SHOULD NOT / MAY), stable IDs
  (REQ-NNN, DEC-NNN, ADR-NNNN, P1â€“P5, T1â€“T16, US-N, â€¦), methodology
  names (Saga, TCC, Outbox, Redlock, fencing token, C4, arc42, MADR,
  BIZBOK, 12-Factor, OWASP, AIP, AAA, Expand â†’ Contract, DiÃ¡taxis),
  code identifiers, file paths, structural labels, and the verbatim
  content of any ``` fenced seed block stay English.
- Adapter slash-command files (Claude + Codex) now explicitly point
  to the Output Language Contract so it is re-loaded on every
  `/hx-*` invocation.

### Changed

- **Templates collapsed to single English source.** Removed the
  `.zh.*.j2` / `.en.*.j2` template pairing; each artifact now has one
  canonical template (`playbooks/*.md.j2`, `AGENTS.md.j2`, etc.).
  Eliminates drift between language variants and protects AI
  pattern-match accuracy â€” the agent's contract surface is one
  language by construction.
- Adding a new output language now requires zero scaffold work; only
  add a `Messages` literal in `i18n.py` for CLI runtime strings.

### Removed

- ~22 `.zh.*.j2` template variants (playbooks, principle pack, evals
  README, AGENTS.md, CLAUDE.md, slash commands, scripts, CI / TOML /
  gitignore).
- `ScaffoldEngine.render_localized()` â€” no remaining callers; the
  public scaffold API is now `render_file` / `render_static` /
  `ensure_dir` only.
- `tests/templates/test_localization_parity.py` and
  `tests/templates/test_seed_contract.py` â€” both assumed zh/en
  template pairs that no longer exist.
- Per-locale skill-description dicts in claude / codex adapters
  (`_SKILL_DESCRIPTIONS_ZH`).

### Migration notes

- Projects scaffolded with 0.1.0 keep their existing files. To pick
  up the new English templates + Output Language Contract, run
  `harness init --force` in the project root.
- The `--output-lang` flag and `.harness/config.toml#output_lang`
  field are unchanged on the wire. What changed is *when* they're
  consumed: previously at scaffold time, now at `/hx-*` runtime.

## [0.1.0] - 2026-06-07

Initial public release on PyPI as **`harness-scaffold`**.

### Added

- `harness init` command â€” scaffold a new harness-managed repo with
  `.harness/` brain, per-agent adapters, and shared CI/pre-commit
  configuration. Supports `--profile`, `--flow`, `--lang`,
  `--output-lang`, `--preset`, `--agent` flags.
- `harness doctor` command â€” self-check for rule/guardrail sync,
  drift, and dead rules.
- 14 tool-neutral playbooks covering the SDD cycle: constitution,
  baseline, next, propose, clarify, design, plan, tasks, analyze,
  implement, verify, review, archive, doctor.
- Adapter registry pattern. Bundled adapters: Claude Code, Codex
  CLI. Cursor / Copilot / Gemini stubs in the registry for future
  extension.
- Bilingual templates (zh / en) with a structural-parity test
  (`tests/templates/test_localization_parity.py`) that prevents
  drift in headings, frontmatter, and Provenance blocks.
- Single deterministic `verify` script shared by hooks, pre-commit,
  and CI.
- Test suite: 130 tests including 60 localization-parity assertions.
- GitHub Actions CI: matrix over Python 3.10â€“3.13, with ruff,
  ruff format, mypy strict, and pytest + coverage.
- `.github/workflows/release.yml` â€” builds sdist + wheel on tag push,
  verifies the tag matches `pyproject.toml`, runs `twine check`,
  publishes to PyPI via trusted publishing, and attaches dists to the
  GitHub release.
- `[project.urls]`, `keywords`, and PEP 639 classifier set in
  `pyproject.toml` for richer PyPI metadata.
- Install paths documented for both PyPI (`pip install
  harness-scaffold`) and `git+https://github.com/Onebyte1943/harness-scaffold.git`.

### Notes

- PyPI distribution name is **`harness-scaffold`** because the bare
  `harness` name was already taken on PyPI by an unrelated project.
  `[tool.uv.build-backend]` pins `module-name = "harness"` so the
  wheel still ships `src/harness/` and `import harness` /
  `harness <cmd>` continue to work unchanged.
- `importlib.metadata.version()` looks up the distribution name
  `harness-scaffold`, not the module name; `harness --version` reports
  `0.1.0` (not `0.0.0+unknown`).
- Minor robustness fixes in `harness init` and `ScaffoldEngine` â€”
  narrower exception handling around `questionary`, `sys.stdin`, and
  Jinja `TemplateNotFound`.
